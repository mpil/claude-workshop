1System Overview

The Inventory Management System (IMS) is a two-tier web application. A browser-hosted Vue 3 single-page application communicates over HTTP/JSON with a Python FastAPI server. All data is held in-memory — there is no database layer. Data resets to its initial state whenever the server process restarts.

Current-state limitation: The absence of a persistence layer means any changes made at runtime (purchase orders, tasks) are lost on server restart. This is a known gap flagged in the original vendor handoff and addressed in the RFP requirements.

2Frontend Architecture

The frontend is a Vue 3 single-page application bundled with Vite. Application entry is main.js, which creates the router and mounts the root App.vue component onto #app.

Composables (Shared State)

Vue composables in this codebase follow the singleton pattern: reactive ref() values are declared at module scope (outside the exported function), so all consumers share the same reactive state across the entire application lifetime. Re-calling useFilters() in a new component does not create a fresh instance — it returns refs that point to the same underlying reactive values.
Composable File Singleton state held Purpose
useFilters composables/useFilters.js selectedPeriod, selectedLocation, selectedCategory, selectedStatus Global filter state shared by FilterBar and all views. Exposes getCurrentFilters() which maps UI values to API query params (e.g. selectedLocation → warehouse).
useI18n composables/useI18n.js currentLocale (persisted to localStorage), currentCurrency Locale switching (English / Japanese). Provides t(key) translation function with English fallback, plus domain-specific translators for product names, warehouse names, and customer names.
useAuth composables/useAuth.js currentUser (computed, locale-aware), isAuthenticated Mock authentication. User profile and task list adapt to the active locale. No real token/session management.

3Backend Architecture

The backend is a single-file FastAPI application (server/main.py) served by uvicorn on port 8001. All business data lives in Python lists imported from mock_data.py at startup. Two utility functions handle filtering: apply_filters() (warehouse, category, status) and filter_by_month() (direct month match e.g. 2025-01 or quarter match e.g. Q1-2025).

CORS is configured with allow_origins=["*"]. This is appropriate for local development but must be restricted to the application's specific origin before any production deployment.

Filter Utility Functions

FunctionSignatureLogic
apply_filters (items, warehouse?, category?, status?) Filters sequentially. Skips a dimension when value is None or "all". Category and status comparisons are case-insensitive.
filter_by_month (items, month?) Accepts a direct month string (e.g. 2025-03) or a quarter (e.g. Q1-2025). Matches against each item's order_date field. Returns items unchanged when month is None or "all".

4Data Flow

Filter state flows in one direction: from the user interacting with FilterBar, through the shared singleton useFilters composable, to a watch() in each view, which fires an API request and renders the result.

Key naming mismatch: The UI exposes a "Location" filter (selectedLocation), but getCurrentFilters() maps this to warehouse before sending to the API. Similarly, "Period" in the UI becomes month in the API query string. Developers working across both layers should be aware of this translation.
