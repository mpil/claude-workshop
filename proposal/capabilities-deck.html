The Situation

A working system that can't be changed safely

Defects

Reports module — 8+ known issues

Filter wiring incomplete, internationalization gaps, inconsistent API patterns, and console noise left unresolved when the previous vendor's contract ended in November 2024. The issues were logged but never fixed.

Blocked

IT has blocked all changes

No automated test coverage means Meridian IT cannot safely approve any change to the codebase. Every other requirement — Restocking, i18n, UI work — is gated behind this. R3 isn't just a nice-to-have; it unblocks everything else.

Missing

Two capability gaps

Operations team (led by VP R. Tanaka) has no restocking visibility — purchase order decisions are manual and error-prone. Tokyo warehouse staff (12 people) work in English-only views despite variable English proficiency.

Our Approach

Stabilize first. Then extend.

We don't build new features on a broken foundation. Testing infrastructure goes in during week one — before we touch any production code. Every fix and every feature is covered by automated tests as it's built, not retrofitted at the end.

By the time we deliver the Restocking view, the Reports module is already clean and covered. When we hand off, Meridian IT receives a working CI pipeline that runs on schedule and on every pull request — not a test suite that requires integration work on their end.

The previous vendor's handoff documentation is thin. We treat it as a starting point and conduct our own codebase audit in Phase 1 before committing to a final defect list.

R1 — Required

Reports Module Remediation

Audit First

We don't assume the list is complete

The RFP documents at least 8 defects. Our experience with thin vendor handoffs is that undocumented issues always surface during a real audit. We conduct a structured code review before writing a single fix — and share the full findings with Meridian at the end of Phase 1.

Filter Wiring

All four filters, fully connected

Time Period, Warehouse, Category, and Order Status must pass correctly via query params to the FastAPI backend and reflect accurately in the displayed data. We verify each filter end-to-end — both the request path and the rendered result — before closing the issue.

i18n

No hardcoded English strings

Every user-facing string in the Reports view will be routed through the existing internationalization framework. This is prerequisite work for D2 — if we fix Reports properly now, the i18n extension phase is clean and additive rather than corrective.

Patterns

Consistent Vue 3 Composition API

The previous vendor left Reports components using the older Options API while the rest of the app was migrated to Composition API. We will complete that migration in Reports, bringing it in line with established patterns and making future maintenance straightforward.

R2 — Required

Restocking Recommendations View

Input

Operator-supplied budget ceiling

The operations team enters a budget ceiling. The system takes it from there — no spreadsheet, no manual cross-referencing of stock levels and demand data. A simple input that produces a clear, actionable output.

Logic

Deficit × demand × budget ceiling

Items below their reorder threshold are flagged. These are ranked by demand forecast — high-velocity items come first. The ranked list is then clipped to the operator's budget ceiling, so the most critical orders are always selected first. Logic runs server-side (FastAPI) for independent testability.

Output

Ranked purchase order list

Highest-priority items displayed first with quantity, estimated cost, and demand signal. The view is built as a standard Vue 3 component consistent with the rest of the application, accessible from main navigation. Uses existing /api/inventory and /api/demand — no new infrastructure needed.

Optional Enhancements

D1 · Optional add-on

UI Modernization

Visual refresh aligned to Meridian's confirmed design system. We have one open question before contract execution: which design system or component library Meridian prefers. Our approach accommodates any reasonable answer — we just need confirmation at kickoff to avoid mid-engagement rework. The core engagement does not depend on D1.

D2 · Included in core scope

i18n Extension

Extends the existing internationalization framework to remaining modules. Tokyo warehouse views are the priority — stock levels, orders, and any Reports views used by that team. We extend rather than replace the existing setup. R1 lays the groundwork: if we fix i18n gaps in Reports correctly, D2 is clean additive work. Delivered in Phase 4.

D3 · Optional add-on (bundled with D1)

Dark Mode

Operator-selectable theme, persisted across sessions. Several Meridian warehouse floor stations run in low-light environments where dark mode meaningfully reduces eye strain during long shifts. Dark mode is bundled with D1 because implementing it via CSS custom properties during the initial UI pass costs a fraction of retrofitting it later. Separating D1 and D3 would create unnecessary rework.
